# Security Guide

이 문서는 이 저장소에서 자격증명, 세션, 실데이터 검증을 다룰 때의 기본 원칙을 정리합니다.

## 1. 자격증명 저장

지원하는 인증 입력은 두 가지입니다.

- 환경 변수
- 저장 로그인

저장 로그인 사용 시:

- 아이디는 로컬 프로필 파일에 저장됩니다.
- 비밀번호는 Windows Credential Manager 에 저장됩니다.
- 서비스별 세션은 로컬 세션 파일에 저장됩니다.

## 2. 세션 파일

기본적으로 세션은 서비스별로 분리됩니다.

- LMS: `state/lms-session.json`
- MSI: `state/msi-session.json`
- UCheck: `state/ucheck-session.json`
- Library: `state/library-session.json`

이 파일들은 인증된 세션 상태를 담을 수 있으므로 외부 공유나 커밋 대상이 아닙니다.

## 3. 절대 커밋하지 말아야 할 것

- 실제 계정 비밀번호
- 세션 파일
- 로컬 프로필 파일
- Windows Credential Manager 덤프
- 원본 HTML 스냅샷
- 민감한 JSON 응답 원본
- 다운로드한 실제 과제/공지/자료 파일

## 4. 실데이터 검증 시 주의

- 읽기 기능이라도 개인정보가 포함될 수 있습니다.
- 학생 정보, 성적, 학적, 연락처 같은 데이터는 최소한으로만 다룹니다.
- 문서에는 전체 원문보다 핵심 결과만 요약해서 남깁니다.

## 5. 쓰기 기능 안전장치

현재 LMS와 Library 쓰기 기능은 승인 흐름을 사용합니다.

- `confirm=true`
- `approvalToken`
- 같은 세션 재호출

이 장치는 잘못된 제출, 삭제, 수정 실행을 줄이기 위한 것입니다.

원칙:

- 쓰기 tool은 항상 명시적 호출이어야 합니다.
- 애매한 강의 식별 상태에서는 자동 실행하지 않는 편이 좋습니다.
- 삭제 기능은 제출 기능보다 더 보수적으로 다룹니다.
- 도서관 예약은 동행자 이름/학번이 실제 내부 사용자로 해석되는지 먼저 확인하는 편이 안전합니다.

## 6. 문서 보안 원칙

공개 문서에는 다음만 남기는 것을 권장합니다.

- 기능 범위
- 입력/출력 스펙
- 마스킹된 식별자 또는 일반화된 예시
- 재현 가능한 명령

공개 문서에 넣지 말아야 하는 것:

- 실계정 비밀번호
- 민감한 스냅샷 전문
- 장문의 원본 HTML / JSON
- 재사용 가능한 세션 정보

## 7. 운영 상 권장사항

- 테스트용 대표 과목과 대표 과제를 정해 반복 검증합니다.
- 도서관 예약 검증은 가능하면 실제 예약 생성 대신 수정 후 원복 같은 보수적 시나리오를 우선합니다.
- 실데이터 쓰기 검증 후에는 가능하면 원래 상태로 복구합니다.
- 공유 PC 에서는 저장 로그인보다 환경 변수 또는 일회성 로그인 확인이 더 안전할 수 있습니다.
- 세션이 의심스러우면 먼저 세션 파일을 지우고 다시 로그인합니다.

## 8. 브라우저 자동화 사용 원칙

기본 원칙은 HTTP 우선입니다.

Playwright 같은 브라우저 자동화는:

- 구조 파악이 막힐 때
- 모달 / 재인증 / JS 흐름 식별이 필요할 때

에만 보조적으로 사용하는 것이 좋습니다.

## 9. 책임 범위

이 저장소는 대학 서비스의 비공식 자동화 계층입니다.

따라서 다음을 항상 염두에 둬야 합니다.

- 서비스 구조는 언제든 변경될 수 있습니다.
- 자동화는 최소 권한과 최소 범위로 유지하는 편이 좋습니다.
- 공개 저장소 문서는 구현 설명 중심으로 유지하고, 실사용 데이터는 최소화해야 합니다.
